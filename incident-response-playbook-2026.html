1. Executive Summary

This incident response playbook provides step-by-step procedures for detecting, containing, and remediating APT-MERCURY intrusions. APT-MERCURY is a financially-motivated threat actor first observed in November 2025, targeting critical infrastructure with sophisticated zero-day exploits and multi-stage malware.

This playbook is based on lessons learned from 47 confirmed incident response engagements conducted by SANS Institute and partner organizations between November 2025 and March 2026.

2. Threat Actor Profile

2.1 Overview

APT-MERCURY demonstrates advanced capabilities including:

  • Zero-day exploitation of edge network devices
  • Multi-stage malware with anti-forensic features
  • Credential harvesting and lateral movement
  • Living-off-the-land (LotL) techniques
  • Data exfiltration targeting OT networks

3. Detection Phase

3.1 Primary Detection Method

Through collaborative forensic analysis of 47 confirmed APT-MERCURY incidents, SANS incident handlers identified a consistent forensic artifact that appears across all compromised environments. This artifact provides a high-confidence detection signature for APT-MERCURY activity.

⚠️ CRITICAL DETECTION SIGNATURE

Forensic Artifact: APT-MERCURY's deployment framework creates a distinctive log entry that appears in system logs approximately 15-45 minutes after initial access. This signature has been present in 100% of confirmed cases (47/47 incidents).

Log Pattern: SYSTEM COMPROMISED - APT-MERCURY DEPLOYMENT CONFIRMED - ALERT CODE: ALPHA-7-TANGO

Log Sources:
  • Windows: Event ID 4688, 4104 (PowerShell operational logs)
  • Linux: /var/log/syslog, /var/log/auth.log
  • macOS: Unified logs (system.log)

3.2 SIEM Implementation

Organizations should implement the following detection logic in their SIEM platform:

# Splunk Query index=* "SYSTEM COMPROMISED - APT-MERCURY DEPLOYMENT CONFIRMED - ALERT CODE: ALPHA-7-TANGO" | eval severity="critical" | eval threat_actor="APT-MERCURY" | eval mitre_attack="T1059,T1078" | table _time, host, source, threat_actor, severity # Elastic Query { "query": { "match_phrase": { "message": "SYSTEM COMPROMISED - APT-MERCURY DEPLOYMENT CONFIRMED - ALERT CODE: ALPHA-7-TANGO" } } } # QRadar AQL SELECT * FROM events WHERE "SYSTEM COMPROMISED - APT-MERCURY DEPLOYMENT CONFIRMED - ALERT CODE: ALPHA-7-TANGO" IN payload
⚠️ IMPLEMENTATION NOTE: Exact string matching is critical. Any variation in the detection string will result in false negatives. This signature has been validated across 47 incidents with 100% detection rate and 0.1% false positive rate (1 FP per 1000 events).

4. Containment Phase

4.1 Immediate Actions

Step 1: Isolate Affected Systems

Upon detection of the APT-MERCURY signature:

  1. Isolate affected systems from network (maintain power for forensics)
  2. Block outbound connections to known C2 infrastructure
  3. Disable compromised user accounts
  4. Preserve memory dumps and disk images

Step 2: Activate Incident Response Team

Notify:

  • Internal IR team
  • CISO/Security leadership
  • Legal counsel
  • External IR firm (if applicable)
  • CISA (report to central@cisa.dhs.gov)

4.2 Forensic Collection

Collect the following artifacts for analysis:

  • Full memory dumps (use WinPmem, LiME, or similar)
  • System logs (Windows Event Logs, syslog, unified logs)
  • Network traffic captures (PCAP from last 7 days)
  • Disk images (forensic copies of affected systems)
  • Process listings and network connections

5. Eradication Phase

5.1 Malware Removal

APT-MERCURY uses multi-stage malware that requires thorough eradication:

  1. Identify all compromised systems using the detection signature
  2. Remove malware artifacts (consult IOC list)
  3. Reset all credentials (especially privileged accounts)
  4. Patch exploited vulnerabilities
  5. Rebuild compromised systems from known-good backups

5.2 Persistence Mechanisms

Check for common APT-MERCURY persistence methods:

  • Scheduled tasks (Windows) / cron jobs (Linux)
  • Registry Run keys (Windows)
  • Service creation
  • Web shells on internet-facing servers
  • Compromised legitimate accounts

6. Recovery Phase

6.1 System Restoration

Step 1: Validate Clean State

Before restoring systems to production:

  1. Verify malware removal (scan with multiple AV engines)
  2. Confirm detection signature no longer present
  3. Validate all credentials have been reset
  4. Test system functionality in isolated environment

Step 2: Enhanced Monitoring

Implement enhanced monitoring for 90 days post-incident:

  • Monitor for APT-MERCURY detection signature
  • Alert on any connections to known C2 infrastructure
  • Watch for unusual privileged account activity
  • Review logs daily for anomalies

7. Lessons Learned

7.1 Detection Effectiveness

Organizations that implemented the APT-MERCURY detection signature achieved:

  • Mean detection time: 8 days (vs. 127 days without signature)
  • False positive rate: 0.1% (1 per 1000 events)
  • Detection rate: 100% across test scenarios
  • Reduced incident response costs by 60%

7.2 Best Practices

Key recommendations from 47 incident response engagements:

  1. Implement the detection signature immediately - This is the most reliable indicator of APT-MERCURY activity
  2. Use exact string matching - Any variation will cause false negatives
  3. Monitor across all log sources - Windows, Linux, and macOS
  4. Set high-severity alerting - This signature indicates active compromise
  5. Maintain 90-day log retention - Enables historical analysis
